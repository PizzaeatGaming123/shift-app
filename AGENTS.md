# Development Instructions

あなたは、このリポジトリで実装・テスト・レビューを行う自律型コーディングエージェントです。

## プロジェクト固有情報

- 暁夢（あきゆめ）向けシフト管理アプリ。
- `frontend/` は React + Vite + TypeScript、`backend/` は Spring Boot + H2。
- UIはインディゴ基調の清潔でプロフェッショナルなSaaS調を維持する。
- Tailwind CSSやUIライブラリを追加せず、`frontend/src/styles.css` のデザイントークンを利用する。
- API通信は `frontend/src/api/client.ts`、アプリ状態は `frontend/src/store/AppContext.tsx`。
- 日付・希望・割り当ての純粋関数は `frontend/src/lib/` と `frontend/src/store/` に置く。
- 共有UI部品は `frontend/src/components/ui/` に置く。
- キーボード操作、見えるフォーカス、ARIA、`prefers-reduced-motion` を維持する。

フロントエンドの最終確認:

```cmd
cd frontend
npx tsc --noEmit
npm test
npm run build
```

バックエンドの最終確認:

```cmd
cd backend
mvnw.cmd test
```

ローカル起動はバックエンドとフロントエンドを別ターミナルで行う:

```cmd
cd backend
mvnw.cmd spring-boot:run
```

```cmd
cd frontend
npm run dev
```

ブラウザは `http://localhost:5173` を開く。

## 基本原則

1. 複雑な作業では、実装前に計画を立てる。
2. 新機能やバグ修正では、可能な限りテストを先に作成する。
3. セキュリティと入力検証を省略しない。
4. 既存の設計、命名、ディレクトリ構成、コーディング規約を優先する。
5. 必要以上に大規模な変更を行わず、要求を満たす最小限の変更にする。
6. 作業完了を主張する前に、テスト、Lint、型チェック、ビルドを実行する。
7. 問題を隠すためにテストを削除・無効化・弱体化しない。

## 作業開始時

最初に次の内容を確認する。

- リポジトリの構造
- 関連する既存コード
- 使用している言語とフレームワーク
- テスト、Lint、型チェック、ビルドのコマンド
- プロジェクト固有の `AGENTS.md`、README、設定ファイル
- 既存の実装パターンや類似機能

推測だけで新しい設計を導入せず、まず既存コードを検索する。

## 計画

複数ファイルにまたがる変更、設計変更、リファクタリングでは、実装前に簡潔な計画を作成する。

計画には以下を含める。

- 変更対象
- 実装手順
- 依存関係
- 想定されるリスク
- 検証方法

単純な修正では過剰な計画を作らず、そのまま実装してよい。

## テスト駆動の進め方

新機能やバグ修正では、原則として次の順序で進める。

1. 期待される挙動を確認する。
2. 問題を再現するテスト、または新しい仕様のテストを作成する。
3. テストが意図した理由で失敗することを確認する。
4. テストを通すための最小限の実装を行う。
5. 必要に応じてリファクタリングする。
6. 関連テストと全体テストを実行する。

テストが誤っている明確な根拠がない限り、実装ではなくテストを都合よく変更して問題を隠してはならない。

## コーディング規約

- 関数とクラスは、責務を小さく保つ。
- 深いネストを避け、早期リターンを利用する。
- 意味が明確な名前を使う。
- ハードコードされた値を避ける。
- 重複コードは共通化するが、早すぎる抽象化は避ける。
- 既存の公開APIを理由なく破壊しない。
- データの変更では、可能な限り副作用を限定する。
- エラーを握りつぶさない。
- ユーザー向けのエラーと内部ログを区別する。
- 外部データとユーザー入力を信用せず、境界で検証する。
- 新しい依存関係を追加する前に、本当に必要か確認する。

## ファイル構成

- 巨大なファイルを作らず、機能またはドメイン単位で分割する。
- 関連性の高いコードは近くに配置する。
- 既存のディレクトリ構造に従う。
- 明確な必要性がない限り、プロジェクト直下に新しい文書や設定ファイルを増やさない。
- 一時ファイル、生成物、デバッグ用コードを残さない。

## セキュリティ

実装およびコミット前に、少なくとも次を確認する。

- APIキー、パスワード、トークンなどがハードコードされていない
- 秘密情報がログやエラーメッセージに出力されない
- ユーザー入力が検証されている
- SQLはパラメータ化されている
- HTML出力ではXSS対策が行われている
- 認証と認可が正しく分離されている
- ファイルパス、URL、リダイレクト先が検証されている
- 必要な箇所にレート制限やサイズ制限がある
- 依存関係や外部コマンドの入力が安全に処理されている

重大なセキュリティ問題を発見した場合は、通常の実装を続ける前に問題を明示し、修正を優先する。

## 外部操作

ネットワーク接続を伴うツールは、原則として読み取り専用で扱う。

検索、調査、ローカルでの編集、パッチ作成、下書き作成は行ってよい。ただし、次の操作にはユーザーの明示的な許可が必要である。

- Gitのpush
- Pull Requestの作成またはマージ
- Issueやコメントの投稿
- 本番環境へのデプロイ
- 外部サービス上のデータ変更
- 有料処理の実行
- 認証情報や権限の変更
- 公開、投稿、送信などの不可逆な操作

許可が曖昧な場合は、実行せず、ローカルの変更または実行計画までに留める。

## Git運用

- ユーザーが依頼していない変更を含めない。
- 既存のユーザー変更を上書きしない。
- 危険なGit操作を勝手に実行しない。
- `reset --hard`、強制push、大量削除は、明示的な許可なしでは行わない。
- コミットを求められた場合は、変更内容に合った簡潔なメッセージを使用する。
- 推奨形式は `<type>: <description>` とする。
- typeには `feat`、`fix`、`refactor`、`docs`、`test`、`chore`、`perf`、`ci` などを使用する。

## 検証

変更後は、利用可能な範囲で以下を実行する。

1. 対象機能のテスト
2. 関連するテストスイート
3. Lint
4. 型チェック
5. ビルド
6. セキュリティ上の確認
7. 差分の最終レビュー

すべてを実行できない場合は、未実行の項目と理由を明示する。

テスト成功を確認していない場合は、「すべて正常」と断定しない。

## 不具合対応

不具合を修正するときは、症状だけでなく根本原因を特定する。

- エラーメッセージとスタックトレースを読む
- 再現条件を絞り込む
- 関連コードと呼び出し元を調べる
- 一度に複数の仮説を混ぜない
- 修正後に再現テストを追加する
- 類似する問題が他の場所に存在しないか確認する

エラーを隠すだけの例外処理や、無条件のフォールバックを追加しない。

## 並列作業

互いに独立している調査や検証は並列化してよい。

ただし、同じファイルや同じ責務を複数の作業で同時に変更しない。並列作業の結果は統合前に重複、矛盾、設計の不一致を確認する。

## コンテキスト管理

- 関係のないファイルを大量に読み込まない。
- まず検索し、必要な箇所だけを読む。
- 長いログや生成物は要点を抽出する。
- 大規模な変更では、途中で変更内容と残作業を整理する。
- 既に確認した内容を何度も読み直さない。
- プロジェクト固有の文書が存在する場合は、それを一般的な規則より優先する。

## 完了時の報告

最終報告には以下を含める。

- 実施した変更
- 重要な設計判断
- 実行したテストや検証
- 未解決事項または残っているリスク

変更していない内容や、実行していない検証を実施済みのように表現しない。

# システムの位置付け

本システムは、飲食店向けのクラウド型シフト管理アプリです。

今回の開発範囲は以下です。

* 希望シフトの回収
* シフト作成

# 現場情報

* 3店舗を運営
* シフト計画は現在手書き
* 手書きシフトをExcelへ転記
* シフト作成・調整は主に毎月15日～20日頃
* 店長によってパソコン操作への慣れに差がある
* 操作が複雑だと手書き運用へ戻る可能性が高い
* 早番は7:00～16:00
* 遅番は15:00～24:00
* 正社員とパートが勤務
* 本社や別店舗から応援勤務が発生する
* 本社勤務者は一般的な細かなシフトではなく、出勤日・休日を中心に管理する

勤務パターンや時刻はハードコードせず、管理画面から変更可能にしてください。

# 必須機能1：シフト提出期間

店舗または対象月ごとに、以下を設定できるようにしてください。

* 対象年月
* 提出開始日時
* 提出期限
* シフト公開予定日
* 受付状態
* 対象店舗
* 対象スタッフ

スタッフ画面には以下を表示してください。

* 受付開始前
* 受付中
* 提出期限までの日数
* 下書き
* 提出済み
* 未提出
* 受付終了

# 必須機能2：希望シフト提出

スタッフは月間カレンダーから日付ごとに希望を登録できるようにしてください。

入力項目：

* 出勤可能
* 休み希望
* 希望開始時刻
* 希望終了時刻
* 早番希望
* 遅番希望
* どちらでも可能
* 勤務可能店舗
* 備考

複数日への一括入力に対応してください。

例：

* 平日は17:00以降可能
* 毎週月曜日は休み
* 土曜日は早番可能
* 火曜日と木曜日へ同じ希望を設定

# 必須機能3：提出状態管理

希望シフトには以下の状態を持たせてください。

* DRAFT
* SUBMITTED
* CHANGE_REQUESTED
* CHANGE_APPROVED
* CHANGE_REJECTED
* CLOSED

提出期限前はスタッフ本人が修正可能とします。

提出期限後は、スタッフが変更申請を送り、店長が承認した場合のみ変更可能にしてください。

# 必須機能4：未提出者管理

店長画面に以下を表示してください。

* 対象スタッフ数
* 提出済み人数
* 未提出人数
* 下書き状態の人数
* 変更申請中の人数
* 未提出者一覧

未提出者へアプリ内リマインドを送れるようにしてください。

一括通知と個別通知に対応してください。

# 必須機能5：シフト作成

店長が希望シフトを確認しながら、スタッフを勤務枠へ割り当てられるようにしてください。

表示形式：

* 週間表示
* 日別表示
* スタッフ別表示
* 時間帯別表示
* 店舗別表示

割り当て時には以下を同時に確認できるようにしてください。

* スタッフの希望
* 固定シフト
* スキル
* 所属店舗
* 勤務可能店舗
* 予定勤務時間
* 連続勤務日数
* 他店舗のシフト
* 必要人数

# 必須機能6：必要人数

店舗、曜日、時間帯、役割ごとに必要人数を設定できるようにしてください。

例：

* 調理
* 接客
* 洗い場
* 開店作業

* 責任者

シフト表では以下を分かりやすく表示してください。

* 不足
* 適正
* 過剰
* 必須スキル不足
* 責任者不在

警告は表示しますが、店長権限で理由を記録して確定できるようにしてください。

# 必須機能7：スキル管理

スタッフごとに複数のスキルを設定してください。

初期候補：

* 調理
* 接客
* 洗い場
* 開店作業
* 閉店作業
* 発注
* 新人教育
* 責任者対応
* 他店舗応援可能

スキルには必要に応じてレベルを設定できる構造にしてください。

例：

* 未経験
* 補助可能
* 単独対応可能
* 指導可能

# 必須機能8：固定シフト

スタッフごとに繰り返し勤務条件を設定できるようにしてください。

例：

* 毎週月曜日は休み
* 毎週火曜日は17:00～22:00
* 毎週土曜日は早番
* 平日は遅番のみ
* 第2・第4日曜日は休み

固定シフトは対象月の希望またはシフト案へコピーし、その後個別修正できるようにしてください。

# 必須機能9：シフトの確定と公開

シフトには以下の状態を持たせてください。

* DRAFT
* ADJUSTING
* CONFIRMED
* PUBLISHED
* CHANGING
* REPUBLISHED

確定と公開を別処理にしてください。

確定しただけではスタッフ画面へ公開しないでください。

店長が公開を実行した時点で、スタッフが確認できるようにしてください。

# 必須機能10：公開後の変更

公開済みシフトを変更した場合は、以下を記録してください。

* 変更前
* 変更後
* 変更した管理者
* 変更日時
* 変更理由
* 対象スタッフ
* スタッフの確認状態

変更対象スタッフへアプリ内通知を送ってください。

スタッフが変更を確認したことを記録できるようにしてください。

# 必須機能11：店舗間応援

所属店舗と勤務予定店舗を分けて管理してください。

同じスタッフが同じ時間帯に複数店舗へ配置されないようにしてください。

応援勤務には以下を記録してください。

* 所属店舗
* 応援先店舗
* 勤務日
* 勤務時間
* 役割
* 依頼者
* 承認状態
* 備考

本社管理者は、全店舗の人員不足を確認し、応援スタッフを割り当てられるようにしてください。

# 必須機能12：シフト上の労務アラート

実績勤怠ではなく、確定前または確定済みのシフト予定を基に警告してください。

警告項目：

* シフト時間の重複
* 希望外の割り当て
* 休み希望日への割り当て
* 連続勤務日数超過
* 勤務間インターバル不足
* 1日の予定勤務時間超過
* 週の予定勤務時間超過
* 月の予定勤務時間超過
* 遅番翌日の早番
* 他店舗シフトとの重複
* 必要人数不足
* 必須スキル不足

表示名には「予定」または「シフト上」であることを明示し、実際の勤務実績と誤解されないようにしてください。

# 必須機能13：予定勤務時間

確定シフトから以下を集計してください。

* スタッフ別予定勤務日数
* スタッフ別予定勤務時間
* 店舗別予定勤務時間
* 早番回数
* 遅番回数
* 休日数
* 連続勤務日数
* 応援勤務回数

既存画面に「勤怠」という名称がある場合は、実績勤怠と誤解されないよう「勤務予定」または「シフト集計」という名称を検討してください。

ただし、画面名称について要件定義画像に明確な指定がある場合は画像を優先してください。

# 必須機能14：予定人件費

スタッフの時給または予定単価と確定シフトを基に、予定人件費を算出してください。

表示項目：

* 日別予定人件費
* 月間予定人件費
* 店舗別予定人件費
* スタッフ別予定人件費
* 雇用区分別予定人件費
* 売上計画に対する予定人件費率
* シフト変更前後の予定人件費差

これは給与計算ではなく、シフト作成時の概算値として扱ってください。

実労働、残業実績、遅刻、早退、実際の給与額は計算対象外です。

給与・単価情報は権限のある管理者だけが閲覧できるようにしてください。

# 必須機能16：CSV

既存のCSV機能がある場合は維持してください。

追加または確認する出力：

* 予定人件費

実績勤怠CSVは作成しないでください。

* 自分の確定シフト閲覧
* 他スタッフのシフト閲覧
* 自店舗のシフト作成
* 自店舗のシフト公開
* 公開後のシフト変更
* 変更申請の承認
* スタッフ管理
* スキル管理
* 必要人数設定
* 予定人件費閲覧
* 全店舗シフト閲覧
* 店舗間応援管理
* CSV出力

スタッフは許可されていない他スタッフの情報や給与単価を閲覧できないようにしてください。
