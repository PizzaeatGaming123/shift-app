# スマホシフト提出・中番完全削除 設計仕様

## 目的

`要件定義/参考UI１.png`〜`参考UI4.png`、`シフト設定.png`、`シフトパターン.png`、
`モデルシフト.png`、`ランク.png`を最優先資料として、暁夢の運用に合うシフト管理画面を実装する。

最重要成果は、スタッフがスマートフォンから提出期間を選び、各日の勤務可否・勤務時間・メモを入力し、
確認後に一括提出できることである。QRコード、LINE登録、LINE通知連携は対象外とする。

暁夢には中番が存在しないため、画面上の非表示ではなく、フロントエンド、バックエンド、DB、
初期データ、テストから中番を完全に削除する。

## 資料優先順位

1. 今回追加された個別画面画像
2. `参考UI１.png`〜`参考UI4.png`
3. 既存の `300.pdf` と `start_manual.pdf`
4. 要件定義内の文書
5. 既存実装

資料と既存実装が食い違う場合は資料を採用する。資料に存在しない装飾、絵文字、大きな角丸、
グラデーション、カードの多用は追加しない。

## 対象範囲

### スタッフ向けスマホ画面

- シフト提出の入口
- 提出期間選択
- 日付ごとの勤務可否入力
- 開始時刻・終了時刻入力
- 日別勤務メモ入力
- 入力内容確認
- 一括提出
- 提出完了
- 提出済み内容の再編集・再提出
- 確定シフト確認

### 店長向け管理画面

- シフト設定
- シフトパターン
- モデルシフト
- ランク設定
- 中番削除に伴うシフト表、集計、絞り込み、設定画面の更新

### 対象外

- QRコードの生成・読み取り・表示
- LINEアカウント登録
- LINE Login
- LINE Messaging APIによる通知
- メール招待
- スタッフ登録フローの全面再設計

## 中番完全削除

### ドメイン

勤務区分は次の3種類のみとする。

- `early`: 早番
- `late`: 遅番
- `off`: 休み

確定シフトに割り当て可能な勤務区分は `early` と `late` のみとする。

スタッフ提出では、固定区分の選択に加えて開始時刻・終了時刻を保存する。時間入力から自動的に
中番を生成することはない。

### 削除対象

- TypeScriptの `mid`
- Javaの `MID`
- H2の列挙値または制約に含まれる `MID`
- 「中番」という表示文言
- 中番の色、凡例、ボタン、フィルター
- 中番の初期データ
- 中番を前提とする集計ロジック
- 中番を前提とするテスト

### 既存データ

開発用H2データは再生成し、中番データを破棄する。中番を早番または遅番へ自動変換しない。

将来、永続DBへ移行して中番データが存在する場合は、移行前に対象件数を検査し、手動判断用の
移行手順を別途作成する。本実装では無言の変換やフォールバックを行わない。

## スマホ提出フロー

### 1. 提出入口

スタッフログイン後の初期画面は、資料のスマホ画面に合わせた縦長の単一カラムとする。

表示内容:

- 店舗名
- スタッフ名
- 提出対象期間
- シフト提出ボタン
- 確定シフト確認ボタン
- 提出状態

チャット吹き出し、端末モック、絵文字を主役にしない。通常のスマホ業務画面として再現する。

### 2. 提出期間選択

店長が設定した回収期間を選択する。選択肢には開始日、終了日、提出期限、提出状態を表示する。

利用可能な期間が1件だけなら初期選択する。利用可能な期間がない場合は入力画面へ進めず、
「現在提出できるシフトはありません」と表示する。

### 3. 日別入力

選択期間の日付を縦方向の一覧で表示する。月間カレンダーから日付を1件ずつ開く方式は廃止する。

各日には次を表示する。

- 日付
- 曜日
- 勤務可能／休み
- 開始時刻
- 終了時刻
- 勤務メモ
- 入力エラー

勤務可能を選択した場合だけ開始・終了時刻を有効にする。休みの場合は時刻を無効化して保存しない。

資料の入力密度に合わせ、各日を巨大なカードにしない。日付ごとの境界は細い罫線で表現する。

### 4. 時刻入力

- 24時間表記
- 店舗営業時間内
- 開始時刻は終了時刻より前
- 設定された刻み幅で選択
- 初期値はシフトパターンまたは直前の提出値

早番・遅番のパターンを選ぶと開始・終了時刻を反映できる。自由入力が許可されている場合は、
パターン適用後の時刻を変更できる。

### 5. メモ

日ごとに200文字以内の勤務メモを入力できる。資料にある「変わってくれませんか？」などの連絡を
想定する。空欄を許可する。

### 6. 確認画面

提出前に次を一覧表示する。

- 勤務可能日と時間
- 休みの日
- メモ
- 未入力または不正な日

エラーがある場合は提出ボタンを無効化し、対象日へ戻れるようにする。

### 7. 一括提出

確認画面の「シフト提出」で期間内の内容を一括保存する。

保存は期間単位のAPIで行い、全件成功または全件失敗とする。途中まで保存された状態を作らない。
二重タップ中はボタンを無効化する。

### 8. 完了画面

成功後は次を表示する。

- 提出完了
- 対象期間
- 提出日時
- 提出内容を確認するボタン
- 修正するボタン

完了画面から同じ期間を再度開き、修正後に再提出できる。

## スタッフ提出データ

現行の1日1区分だけを保存する `ShiftRequest` は、時間範囲を保持できる形へ拡張する。

想定モデル:

```text
ShiftSubmission
- id
- staffId
- storeId
- periodStart
- periodEnd
- submittedAt
- status

ShiftSubmissionDay
- id
- submissionId
- date
- availability: WORK / OFF
- startTime: nullable
- endTime: nullable
- note
```

同一スタッフ・同一提出期間につき有効な提出は1件とする。再提出は既存提出を更新し、
`submittedAt`を更新する。

店長のシフト表では `WORK` の希望時間を希望シフトとして表示し、`OFF` は休み希望として表示する。

## API

### 取得

```text
GET /api/staff/shift-submission-periods
GET /api/staff/shift-submissions?periodStart=YYYY-MM-DD&periodEnd=YYYY-MM-DD
```

### 一括提出

```text
PUT /api/staff/shift-submissions
```

リクエストには対象期間と全日分の入力を含める。サーバー側で次を検証する。

- ログインスタッフ本人の提出である
- 店舗が一致する
- 対象期間が提出可能
- すべての日付が対象期間内
- 日付が重複していない
- 勤務可能日は開始・終了時刻が必須
- 開始時刻が終了時刻より前
- 営業時間内
- メモが200文字以内
- 中番値が送信されていない

不正な場合は日付と理由を含む400エラーを返す。権限違反は403とする。

## 店長向けシフト設定

`シフト設定.png`を基準に、次を1画面にまとめる。

- シフト回収プレビュー
- シフト周期
- 提出対象期間
- 提出期限
- 初回指定
- 回収期間の追加

プレビューと入力フォームは白背景、薄い罫線、直角に近い外観とする。設定保存後、
スタッフ画面の提出期間選択へ反映する。

## シフトパターン

`シフトパターン.png`を基準に、次を設定する。

- 早番の名称、開始時刻、終了時刻、色
- 遅番の名称、開始時刻、終了時刻、色
- 休み
- 自店舗／ヘルプの表示例
- 未確定表示と確定表示のプレビュー

中番行は表示もデータも存在させない。角丸は1〜2px程度に抑え、希望は点線、確定は実線で表示する。

## モデルシフト

`モデルシフト.png`を基準に、曜日・日付ごとに時間帯と必要人数を設定できる表を使用する。

- 時間帯の追加・削除
- 曜日ごとの必要人数
- 日付別上書き
- シフト表の `配置人数/必要人数` への即時反映

早番・遅番の固定区分だけではなく、任意の時間帯を複数設定できる。中番という固定名称は使用しない。

## ランク設定

`ランク.png`を基準に、ランク一覧とスタッフ保有数を同じ画面で表示する。

- 5段階
- ランク名の編集
- 並び順
- 保有者数
- スタッフへの割り当て

シフト表の日別労働力合計へ反映する。既存の数値1〜5との互換性を維持する。

## UI方針

- 資料画像と同じ情報順序を優先
- 白背景、青い見出し、薄い灰色の罫線
- 原則として角丸なし、必要箇所のみ1〜2px
- 影はモーダルと浮遊メニューに限定
- 絵文字を使用しない
- アイコンは既存資料に存在する場合のみ使用
- タップ領域は44px以上を確保
- 見えるフォーカスを維持
- ラベルと入力の関連付けを行う
- エラーを色だけで伝えない
- `prefers-reduced-motion`を維持
- スマホ幅320pxから利用可能

## エラー処理

- 取得失敗: 再読み込みボタン付きの画面内エラー
- 提出失敗: 入力内容を保持して画面上部へエラー表示
- 日別検証エラー: 対象日にもメッセージ表示
- 通信中: 提出ボタンを無効化
- 二重送信: サーバー側でも同一内容を安全に更新
- 期間終了後: 閲覧は可能、編集・提出は不可

## テスト

### フロントエンド

- 提出入口から期間選択へ進める
- 全日分の入力欄を表示する
- 休み選択で時刻欄が無効になる
- 開始・終了時刻の検証
- メモ文字数検証
- 確認画面の内容
- 一括提出APIの呼び出し
- 失敗時に入力を保持する
- 完了画面と再編集
- 320px、375px、430pxのレイアウト
- 中番の文言・ボタン・型が存在しない

### バックエンド

- 正常な一括提出
- 更新提出
- 対象期間外の日付を拒否
- 重複日付を拒否
- 不正な時刻を拒否
- 営業時間外を拒否
- 文字数超過を拒否
- 他スタッフの提出を拒否
- トランザクションのロールバック
- 中番値を受理しない

### 回帰確認

- 店長シフト表に希望時間と休み希望が表示される
- 一括操作が新しい希望データを利用する
- 確定シフトが表示できる
- 早番・遅番のみで労働時間・人件費・ランク計を算出する
- シフト設定がスタッフ提出期間へ反映される

## 移行順序

1. 中番を除いた新しいドメインモデルとDBを準備
2. 一括提出APIをテスト先行で実装
3. スマホ提出フローを実装
4. 店長シフト表を新しい希望データへ接続
5. シフト設定・パターン・モデルシフト・ランク画面を資料準拠に更新
6. 旧 `ShiftRequest` と中番コードを削除
7. 全テスト、型チェック、ビルド、実ブラウザ確認

## 完了条件

- スマホで期間選択から一括提出完了まで操作できる
- 再編集・再提出できる
- 中番がコード、DB、初期データ、UI、テストに存在しない
- QRコードやLINE連携を実装していない
- 店長画面へ希望時間と休み希望が反映される
- 追加資料4画面が資料と同じ情報構造・密度で表示される
- フロントエンドとバックエンドの全検証が成功する
